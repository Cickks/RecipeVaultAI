| D-001 | Use one Expo/React Native TypeScript codebase for iOS and Android. | Shared product behavior and faster mobile delivery. | Separate Swift/Kotlin apps; Flutter. | Native edge cases still require platform testing; Expo compatibility constrains versions. | Phase 0 | Only if measured platform limitations block core workflows. |

| D-004 | Use React Query for remote/cache state and Zustand for auth/local state. | Separates server lifecycle concerns from app/session state. | Redux; Zustand for everything; React context only. | Requires careful cache keys and clearing on identity changes. | Phase 0/1 | Yes if scale or debugging evidence warrants it. |
| D-005 | Use Supabase Auth, Postgres/PostgREST, migrations, and RLS as the initial backend. | Delivers managed auth and database authorization without a separate API. | Firebase; custom Node/.NET API; local-only persistence. | Public clients can call PostgREST directly, so schema controls and RLS must be complete. | Phase 0/1 | Revisit only if backend workflows outgrow Supabase boundaries. |
| D-006 | Put `user_id` on user-owned rows and enforce owner RLS with `auth.uid()`. | Tenant isolation must be enforced server-side. | Client filtering; opaque per-user schemas. | Policies and indexes require disciplined migration testing. | Phase 1 | No; only policy implementation details may evolve. |
| D-007 | Couple child/join ownership with composite foreign keys. | Prevents same-row ownership claims from referencing another user's parent. | RLS alone; trigger-only validation. | More composite unique constraints and migration complexity. | Phase 1 | No without equivalent proven database enforcement. |
| D-008 | Keep a local sample-data fallback when Supabase public env values are absent. | Supports development and UI work without secrets or hosted access. | Fail app startup; always require Supabase. | Local behavior can hide backend gaps, so phase completion must use real Supabase. | Phase 1 | Remove or narrow before production if it can confuse release builds. |
| D-009 | Use email/password Auth with verification, recovery, and `recipevaultai://` callback routes. | Matches Phase 1 account requirements and Expo mobile deep linking. | Magic links only; social providers; web callback page. | Custom schemes are weaker than verified universal/app links and require transaction correlation. | Phase 1 | Yes; move production auth to verified links when deployment assets exist. |
| D-010 | Keep all real env files ignored; commit only placeholder `.env.example`. | Mobile/public configuration and backend secrets require clear separation. | Commit development env; manual undocumented setup. | Developers must provision local values separately. | Phase 1 | No. |
| D-011 | Treat AI import as a protected backend service with static privileged prompts and delimited/encoded untrusted content. | Prevents provider secrets in the app and user content entering privileged prompt positions. | Direct mobile provider calls; raw prompt concatenation. | Adds backend, rate-limit, validation, and observability work. | Planned Phase 5 | No; provider-specific mechanics can evolve. |
